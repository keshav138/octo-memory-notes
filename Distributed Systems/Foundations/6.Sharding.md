### Definition

**Sharding** is splitting a large dataset into smaller, independent pieces (**shards/partitions**) and distributing them across multiple nodes so the system can **scale writes, storage, and parallel reads**.

---

### Intuition

Instead of one machine holding everything:

```text
All users → Node A  (bottleneck ❌)
```

You distribute:

```text
Users A–M → Node 1
Users N–Z → Node 2
...
```

Each node handles only its slice → better throughput and scalability.

---

### What you gain

- Horizontal scalability (add nodes to grow)
    
- Higher write throughput (writes spread across shards)
    
- Parallel reads
    

### What you pay

- More complex queries (especially cross-shard)
    
- Rebalancing cost when cluster changes
    
- New failure modes (skew, hotspots)
    

---

# 1) Range Partitioning

### Definition

Partition data by **key ranges** (ordered segments).

```text
A–M → Node 1
N–Z → Node 2
```

---

### Why it’s useful

- Preserves **ordering** of keys on each shard
    
- Enables **efficient range scans** (e.g., BETWEEN queries)
    

---

### Example

Usernames or timestamps:

```text
2026-01 → Node 1
2026-02 → Node 2
```

A query like “all orders in Feb” hits one shard.

---

### Use cases

- Time-series data (logs, metrics)
    
- Ordered datasets (IDs with temporal meaning)
    

---

### Trade-offs

- **Hotspots**: if many writes land in one range (e.g., “latest timestamp”), that shard gets overloaded
    
- Requires careful key design (e.g., bucketing timestamps)
    

---

# 2) Hash Partitioning

### Definition

Apply a **hash function** to the partition key and assign based on the hash.

```text
partition = hash(user_id) % N
```

---

### Why it’s useful

- Produces **even distribution** (good load balancing)
    
- Avoids hotspots (keys spread uniformly)
    

---

### Example

```text
user_id = 123 → hash → shard 3
user_id = 987 → hash → shard 1
```

---

### Use cases

- High write throughput systems
    
- Workloads without range queries (key-value access)
    

---

### Trade-offs

- **Destroys locality/order** → range queries become expensive  
    (“give me users between 100–200” now touches many shards)
    
- Cross-shard queries are common
    

---

### Systems

- Apache Cassandra
    
- MongoDB (hashed sharding option)
    

---

# 3) Consistent Hashing

### Definition

A **ring-based hashing scheme** where both nodes and keys are placed on a hash ring. Each key is assigned to the next node clockwise.

---

### Intuition

Instead of `hash(key) % N` (which breaks when N changes), you map everything onto a **circle**:

```text
[0 … 2^32-1] in a ring
```

- Nodes occupy positions on the ring
    
- Keys map to positions and go to the nearest node
    

---

### Key benefit

> When adding/removing a node, only about **1/N of keys move**, not all.

---

### Example

```text
Ring: Node A —— Node B —— Node C

Add Node D:
Only nearby keys shift to D (not entire dataset)
```

---

### Use cases

- Systems with **frequent scaling**
    
- Distributed caches, CDNs
    

---

### Trade-offs

- More complex than simple hashing
    
- Needs techniques like **virtual nodes** to balance load well
    

---

### Systems

- Apache Cassandra
    
- Amazon Dynamo
    
- CDNs
    

---

# 4) Secondary Indexes on Partitioned Data

Once data is sharded, querying by **non-shard keys** becomes hard.

Example:

```text
Shard key = user_id
Query = "find all users with age = 25"
```

Age is not the partition key → data is spread everywhere.

---

## Approach A: Scatter–Gather

### Idea

Query **all shards**, then merge results.

```text
Query → Node 1, Node 2, Node 3 → merge → result
```

---

### Pros

- Simple to implement
    
- No extra index maintenance
    

---

### Cons

- Expensive at scale (fan-out to all nodes)
    
- High latency
    

---

### Used by

- MongoDB (for some queries)
    

---

## Approach B: Term-Based (Global) Index

### Idea

Maintain a **separate index** partitioned by the query attribute.

```text
Index: age → user_ids
```

Now:

```text
age=25 → index → list of user_ids → fetch data
```

---

### Pros

- Efficient lookups
    
- Avoids full-cluster scans
    

---

### Cons

- Extra storage
    
- Write amplification (must update index + data)
    
- Consistency challenges
    

---

### Used by

- Amazon DynamoDB (Global Secondary Indexes)
    

---

# 5) Rebalancing

### Definition

Moving data between nodes when:

- Nodes are added (scale out)
    
- Nodes are removed (failures or scale in)
    

---

### Problem

If partitions are fixed like:

```text
hash(key) % 4
```

Adding a 5th node:

```text
hash(key) % 5
```

→ **almost all keys remap** → massive data movement

---

## Better Approach: Dynamic Partitioning

### Techniques

- **Split partitions** when they grow large
    
- Use **consistent hashing**
    
- Use **many small partitions** instead of few large ones
    

---

### Goal

- Move **minimal data**
    
- Keep cluster balanced
    

---

### Trade-offs

- Background data movement (costly)
    
- Temporary performance impact
    

---

# Final Comparison (Important)

|Method|Strength|Weakness|
|---|---|---|
|Range|Fast range queries|Hotspots|
|Hash|Uniform distribution|Poor range queries|
|Consistent Hashing|Minimal reshuffling|Complexity|
|Scatter–Gather|Simple|Expensive queries|
|Global Index|Fast queries|Write overhead|

---

# Mental Model

- **Sharding = splitting data**
    
- **Replication = copying data**
    

You almost always combine both:

```text
Shard → then replicate each shard
```

---

# One-line Summary

> Sharding distributes data across nodes to scale systems, and different strategies (range, hash, consistent hashing) trade off between query efficiency, load balance, and operational complexity, while indexing and rebalancing determine how usable and maintainable the system remains at scale.
