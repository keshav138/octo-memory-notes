## The essential mental model

Before anything else, burn this into your brain: **a distributed system is a set of computers that look like one computer to the outside world, but can have any subset of those computers fail at any moment, messages can be delayed or lost, and clocks disagree.** Every design decision flows from handling those three realities.

## 1. Theoretical foundations

**CAP theorem** (above) — the starting point for every design conversation. The key insight people miss: partition tolerance is _not optional_ in practice. Networks always partition eventually. So the real choice is between CP and AP during a partition event.

**PACELC theorem** — extends CAP. Even when the network is _healthy_ (no partition), you still trade off latency vs consistency. This is more useful in practice.

**Consistency models** — a spectrum, not a binary:
[[2. Consistency In Distributed Systems]]

- _Linearizability_ — the strongest. Every operation appears instantaneous. Used by Google Spanner, etcd.
- _Sequential consistency_ — all nodes see the same order, but maybe not real-time.
- _Causal consistency_ — causally related operations are seen in order. Used by MongoDB.
- _Eventual consistency_ — given no updates, all nodes converge. Used by DynamoDB, Cassandra.

**FLP impossibility** — in an async network where even one process can crash, no deterministic consensus algorithm can always terminate. This is why real systems use timeouts and randomisation (Raft, Paxos).

## 2. Consensus algorithms

How do distributed nodes agree on a single value?

**Paxos** — the OG algorithm (Lamport, 1989). Notoriously hard to understand. Two phases: _prepare_ and _accept_. A proposer collects a quorum, proposes a value, nodes accept if they haven't seen a higher ballot number.

**Raft** — designed explicitly to be understandable. Has three roles: leader, follower, candidate. Leader election via randomised timeouts. All writes go through the leader, then replicate to followers. Used in etcd, CockroachDB, TiKV.

## 3. Replication

**Why replicate?** Fault tolerance + read scalability.

**Single-leader replication** — all writes go to one node, replicated to followers. Simple but leader is a bottleneck. Used in PostgreSQL streaming replication, MySQL, MongoDB.

**Multi-leader replication** — multiple nodes accept writes. Higher availability, but you get write conflicts. Used in multi-datacenter setups (CockroachDB, Cassandra).

**Leaderless replication (Dynamo-style)** — any node can accept a write. Uses quorums: write to W nodes, read from R nodes, where W + R > N (total replicas). This ensures overlap. Cassandra, DynamoDB, Riak use this.

**Replication lag problems to know:** read-your-writes consistency, monotonic reads, consistent prefix reads.

## 4. Partitioning (sharding)

Splitting data across nodes so no single node holds everything.

**Range partitioning** — partition by key range (A-M on node 1, N-Z on node 2). Allows efficient range scans, but can create hotspots.

**Hash partitioning** — hash the key to determine partition. Distributes evenly but kills range queries. Used by Cassandra, MongoDB.

**Consistent hashing** — a ring-based approach where adding/removing nodes only moves 1/N of keys. Used by Cassandra, Dynamo, CDNs.

**Secondary indexes on partitioned data** — two approaches: scatter-gather (query all partitions, merge results) used by MongoDB; or term-based index (a global index partitioned by term) used by DynamoDB GSIs.

**Rebalancing** — when nodes are added/removed. Avoid "fixed number of partitions" for dynamic clusters; use dynamic partition splitting instead.

## 5. Distributed transactions

This is where it gets hard.

**2-Phase Commit (2PC)** — a coordinator sends _prepare_ to all participants, then commits only if all vote yes. Blocking protocol — if the coordinator crashes after prepare but before commit, participants are stuck waiting forever. Used in traditional databases.

**3-Phase Commit (3PC)** — adds a "pre-commit" phase to allow non-blocking recovery. Rarely used in practice due to complexity.

**Saga pattern** — for microservices. Instead of a distributed transaction, you chain local transactions with compensating transactions for rollback. Used in event-driven architectures. Two flavours: choreography (events trigger next steps) and orchestration (a central coordinator).

**MVCC (Multi-Version Concurrency Control)** — readers never block writers; each transaction sees a snapshot. Used in PostgreSQL, CockroachDB, Spanner.

## 6. Clocks and ordering

You cannot trust clocks in a distributed system.

**Logical clocks (Lamport clocks)** — a counter that increments on every event. If A → B (A happened before B), then `timestamp(A) < timestamp(B)`. But the reverse isn't guaranteed.

**Vector clocks** — a vector of counters, one per node. Can determine causal relationships between any two events. Used by Dynamo, Riak.

**Hybrid Logical Clocks (HLC)** — combine physical and logical time. Used by CockroachDB.

**TrueTime (Google Spanner)** — GPS + atomic clocks across datacenters. Returns a time interval with bounded uncertainty, enabling linearizable transactions globally.

## 7. Message queues & streaming

**Why queues?** Decouple producers from consumers, handle backpressure, enable async processing.

**Kafka** — the dominant log-based message broker. Topics are partitioned and replicated. Consumers track their own offset. Retains messages for configurable duration, enabling replay. Used for event streaming, CDC, log aggregation.

**RabbitMQ** — traditional message queue. Supports complex routing (exchanges, bindings). Messages deleted after consumption (no replay by default).

**Key concepts**: at-most-once, at-least-once, exactly-once delivery; consumer groups; backpressure; log compaction.

## 8. Load balancing & service discovery

**Load balancing algorithms**: round robin, weighted round robin, least connections, consistent hashing (for sticky sessions).

**L4 vs L7**: Layer 4 (TCP) balancing is fast but dumb. Layer 7 (HTTP) can route based on URL, headers, content — used by Nginx, HAProxy, Envoy.

**Service discovery**: how do services find each other in a dynamic environment? Client-side (Eureka, Ribbon) vs server-side (AWS ALB, Consul). DNS-based discovery is simplest.

---

## 9. Distributed caching

**Cache aside** — application checks cache first, loads from DB on miss. Most common pattern.

**Write-through / write-behind** — cache is always updated on write. Reduces read latency, increases write latency (write-through) or risks data loss (write-behind).

**Cache invalidation** — the hard part. TTL-based, event-driven, or version-based.

**Redis** — single-threaded, in-memory, supports data structures. Has replication and Cluster mode (hash slots for sharding). Sentinel for HA without Cluster.

**Memcached** — simpler, multi-threaded, pure caching. No persistence.

## 10. Distributed system patterns to know

- **Circuit Breaker** — stop calling a failing service, allow recovery time (Hystrix, Resilience4j)
- **Bulkhead** — isolate failures to a partition, don't let one slow service drain all threads
- **Sidecar / Service Mesh** — Envoy proxy handles cross-cutting concerns (retries, tracing, mTLS). Istio, Linkerd.
- **CQRS + Event Sourcing** — separate read and write models; store events as source of truth
- **Two-Phase Locking vs MVCC** — know when each is appropriate

## HDFS and MapReduce — what they were, and why they faded

### What is HDFS?

Hadoop Distributed File System is a file system designed to store enormous files (think petabytes) across commodity hardware. It was modeled after Google's GFS paper (2003).

**Key HDFS characteristics:**

- Files split into large blocks (128 MB default)
- Each block replicated 3× across DataNodes (on different racks)
- NameNode holds all metadata _in RAM_ — single point of bottleneck and failure
- Write-once, read-many model — no random writes
- Optimized for sequential scans of large files, not small random I/O

### What is MapReduce?

A programming model for processing massive datasets in parallel. Two phases:

**Map phase** — each mapper reads a block of data and emits key-value pairs. Runs locally on the DataNode holding the data (data locality — avoids network transfer).

**Reduce phase** — the framework sorts and shuffles all values for the same key to one reducer, which aggregates them.

### Why we don't use MapReduce today

This is one of the most important "why" questions in the field.

**Problem 1 — Disk I/O at every step.** MapReduce writes all intermediate data to HDFS between every stage. A 10-stage pipeline touches disk 9 times. This was designed for fault tolerance (you can restart a failed stage from the last checkpoint) but it's brutally slow.

**Problem 2 — Latency is terrible.** The minimum latency for any job is many seconds, often minutes. You cannot do interactive queries or real-time analytics.

**Problem 3 — The NameNode is a bottleneck.** All metadata lives in one node's RAM. For billions of small files, the NameNode runs out of memory. For high-traffic clusters, it becomes a throughput bottleneck.

**Problem 4 — It's hard to express iterative algorithms.** Machine learning algorithms (like gradient descent) need to loop over data many times. In MapReduce, each iteration is a full new job writing to disk. A 100-iteration ML training job = 100 full MapReduce jobs.

**Problem 5 — Real-time data became the norm.** As businesses moved from "batch nightly reports" to "dashboard updates every second," MapReduce's batch model became a liability.

### What replaced it?

**Apache Spark** — keeps data in-memory as RDDs (Resilient Distributed Datasets), then DataFrames/Datasets. A multi-stage pipeline stays in RAM between stages. 10-100× faster than MapReduce for iterative workloads. Has streaming (Spark Streaming, Structured Streaming), ML (MLlib), and SQL (Spark SQL) built in.

**Apache Flink** — true stream-first processing. Treats batch as a special case of streaming. Better for low-latency event processing than Spark.

**The modern data stack**: S3/GCS/Azure Blob as the storage layer (object storage is cheaper and decoupled from compute), with Spark/Dask/Trino/Databricks as the compute layer. This is the **separation of compute and storage** — you don't need to keep expensive compute nodes running just to keep your data warm.

**Cloud data warehouses** (BigQuery, Snowflake, Redshift) handle analytical workloads with columnar storage, vectorized execution, and automatic scaling — no MapReduce jobs needed.

HDFS still exists in on-premise Hadoop clusters, but greenfield projects almost universally start with object storage + cloud-native compute now.
