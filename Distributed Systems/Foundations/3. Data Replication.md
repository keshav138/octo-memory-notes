**Synchronous replication**: Synchronous replication ensures that data is replicated to all replicas before acknowledging the success of a write operation. It guarantees strong consistency but can introduce higher latency and potential performance bottlenecks.

Synchronous Replication is recommended for:

1. **Applications requiring strong consistency**: Synchronous replication is recommended when maintaining strict consistency across replicas is crucial, such as financial systems, stock trading platforms, or critical data storage where any inconsistency can have severe consequences.

2. **Environments with lower write throughput requirements:** Synchronous replication is suitable when the system can tolerate slightly higher latencies for write operations in exchange for strong consistency guarantees.

**Asynchronous replication:** Asynchronous replication allows the primary replica to acknowledge a write operation immediately without waiting for all replicas to replicate the data. Replication to secondary replicas occurs asynchronously, ==providing lower latency and higher write throughput but potentially introducing temporary data inconsistencies during failures or conflicts.==

**Single-leader replication:** In single-leader replication, there is only one designated leader node responsible for handling all write operations. The leader node receives write requests from clients, processes them, and ensures the replication of data to the follower nodes. Follower nodes replicate the data from the leader and typically handle read operations. This type of replication provides strong consistency and ensures that all write operations are serialized and applied in the same order on all replicas. ==One of the drawbacks of the single-leader approach is that the leader becomes a bottleneck impacting the write throughput. In case the leader fails the durability of the data may become a risk.==

**Multi-leader replication:** To overcome the downside of single-leader replication, in multi-leader replication, there are multiple leader nodes, each responsible for handling a subset of write operations. Each leader node has its own set of follower nodes that replicate the data from that specific leader. Write operations can be directed to any of the leader nodes, and the updates are asynchronously propagated to the respective followers. Multi-leader replication provides benefits such as improved write scalability and reduced write latency since write operations can be processed concurrently by different leaders. ==However, managing conflicts and ensuring data consistency across multiple leaders can be more complex, requiring conflict resolution mechanisms and careful coordination.==

**IMP**
In any leader-based approach, the replication method used can be:

**1. Statement-based replication** where the leader logs every write statement it executes and sends the log to its followers. The followers execute the statements in the log to replicate the data. ==This replication breaks down when non-deterministic statements like Now() or Rand() are used in the statement.== Workaround is for the leader to replace any such non-deterministic function with the exact return values when executed at the leader. ==MySQL used this prior to the 5.1 version.==

**2.** **Write-ahead log shipping** where the log is an append-only sequence of bytes of every write to the database. The leader writes this log and ships it to all the followers for data replication. ==Oracle and PostgreSQL use this method. The drawback is that WAL contains low-level details like which bytes were changed on which data block making it difficult for the leader and follower to be on different versions of the database software.==
Tad Confusing so -> [[4. Write-ahead log shipping - Replication Model]]

**3.** **Logical/row-based log replication** relies on separate replication logs than WAL. These logical logs typically will have the row-level details of the writes to the database like inserting and updating the new values of the columns, unique information for deleted rows, etc.

**4.** **Trigger-based replication** where the replication is moved from the database to the application code. Triggers with custom code are registered to capture the changed data either in a separate table or log. The replicas then read this data and replicate.

**Leaderless Replication:** Leaderless replication, also known as decentralized replication or peer-to-peer replication, is a form of data replication where there is no designated leader node responsible for coordinating write operations. Instead, all nodes in the system participate in the replication process and handle both read and write operations. ==Each node acts autonomously and independently, making decisions about data replication and conflict resolution.==
![[Pasted image 20260321005829.png]]

Below are some of the key characteristics and components of leaderless replication:

**Symmetric Replication:** In leaderless replication, all nodes have the same role and responsibilities. Each node maintains a copy of the entire dataset or a partitioned subset of the data. Nodes can handle read-and-write operations locally without relying on a central leader.

**Data Distribution:** Write operations are typically sent to multiple nodes in parallel or broadcast to the entire system. Each node independently processes the write operation and propagates the update to other nodes. The updates are eventually replicated to all nodes, ensuring data consistency across the system.

**Conflict Resolution:** Leaderless replication systems employ conflict resolution mechanisms to handle conflicts that may arise when multiple nodes concurrently update the same data item. Conflict resolution techniques can include ==timestamp-based ordering, vector clocks, conflict-free replicated data types (CRDTs), or other custom resolution strategies==. The goal is to ensure eventual consistency and converge to a consistent state across all nodes.

**Scalability and Fault Tolerance:** Leaderless replication allows for ==easy scalability as new nodes can be added to the system without requiring a centralized coordination mechanism.== It also provides fault tolerance since the absence of a single leader means there is no single point of failure.

Leaderless replication is ==commonly used in distributed databases, peer-to-peer systems, and distributed file systems.== It offers benefits such as high scalability, fault tolerance, and decentralized control. It is ==particularly useful in scenarios where there is a need for high availability, low-latency operations, and the ability to easily scale the system==. ==Amazon’s Dynamo DB, Riak, and Cassandra== are some of the reputed data stores using leaderless replication.

However, leaderless replication introduces ==challenges such as managing conflicts and ensuring data consistency across the nodes==, as well as dealing with scenarios where nodes are temporarily unavailable or network partitions occur. Conflict resolution mechanisms and strategies for handling inconsistencies become critical in ensuring data integrity and convergence.

In summary, data replication is a fundamental concept in distributed computing, enabling systems to achieve high availability, fault tolerance, scalability, and data consistency. Choosing the appropriate replication strategy and configuring replication parameters depend on specific use cases, performance requirements, and consistency needs, striking a balance between data integrity, system performance, and latency.
