Here's your **Dockerfile "Watch Out!" Cheat Sheet** - common trip points and must-know concepts:

## **🔥 THINGS THAT TRIP EVERYONE**

### 1. **Each RUN, COPY, ADD creates a NEW LAYER**
```dockerfile
# ❌ BAD - Creates 3 layers, bloated image
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# ✅ GOOD - Single layer
RUN apt-get update && \
    apt-get install -y curl && \
    rm -rf /var/lib/apt/lists/*
```

### 2. **Order matters for caching**
```dockerfile
# ❌ Copy everything first - cache invalidates if ANY file changes
COPY . .
RUN npm install     # Runs every time ANY file changes

# ✅ Copy package files first, then source
COPY package*.json ./
RUN npm install     # Cached until package.json changes
COPY . .
```

### 3. **WORKDIR creates directory if missing**
```dockerfile
# This works even if /app doesn't exist
WORKDIR /app        # Auto-creates /app
```

### 4. **Difference between CMD forms**
```dockerfile
# Shell form - runs via /bin/sh -c
CMD echo "Hello"    # Can't pass signals (Ctrl+C issues)

# Exec form - runs directly (PREFERRED)
CMD ["echo", "Hello"]   # Handles signals properly

# Common mistake:
CMD ["echo Hello"]   # ❌ WRONG - tries to run "echo Hello" as one command
CMD ["echo", "Hello"] # ✅ CORRECT
```

### 5. **COPY vs ADD - most people overuse ADD**
```dockerfile
# ADD has "magic" features that cause surprises:
ADD https://file.tar.gz /tmp/   # Downloads AND extracts tar?! 
ADD archive.tar.gz ./           # Auto-extracts tar files

# COPY is predictable - just copies
COPY file.txt ./                # Simple copy, no surprises

# Rule: Use COPY unless you need URL download or auto-tar extraction
```

## **🎯 MUST-KNOW CONCEPTS**

### **Context Matters**
```bash
# Docker sends entire directory to daemon (even unused files!)
docker build .

# Context is everything in current directory
# Use .dockerignore to exclude node_modules, .git, etc.
```

### **Each Instruction Runs in a NEW Container**
```dockerfile
# This WON'T work as expected:
RUN cd /app
RUN ls                   # Still in previous directory! Each RUN resets
RUN npm install         

# ✅ Do this instead:
WORKDIR /app
RUN npm install
```

### **Environment Variables - Expansion Rules**
```dockerfile
ENV APP_DIR=/app
WORKDIR $APP_DIR        # Expands to /app

# But escaping behaves differently:
CMD ["echo", "$APP_DIR"]   # Prints "$APP_DIR" (no expansion in exec form)
CMD echo "$APP_DIR"        # Prints "/app" (shell form expands)
```

### **USER switching - root vs non-root**
```dockerfile
FROM ubuntu:22.04
RUN useradd -m appuser
USER appuser           # Switch to appuser

# Everything after USER runs as appuser
RUN mkdir /data        # ❌ PERMISSION DENIED!

# Need to create directories BEFORE switching user
RUN mkdir /data && chown appuser:appuser /data
USER appuser
```

## **🚨 COMMON MISTAKES**

### **1. Forgetting to expose port (or exposing wrong one)**
```dockerfile
EXPOSE 8080   # Documentation only - doesn't actually publish!
```
```bash
# Still need this when running:
docker run -p 3000:8080 myapp  # Host:3000 → Container:8080
```

### **2. Not handling PID 1 properly**
```dockerfile
# ❌ Shell scripts don't forward signals
CMD /start.sh  # Ctrl+C won't stop gracefully

# ✅ Use exec to replace shell
CMD exec /start.sh

# Or use exec form directly
CMD ["/start.sh"]
```

### **3. Installing dev dependencies in production**
```dockerfile
# ❌ BAD
RUN npm install  # Installs everything (dev too)

# ✅ GOOD
RUN npm install --only=production
```

### **4. Hardcoding configs**
```dockerfile
# ❌ BAD
CMD ["node", "app.js", "--db=localhost"]

# ✅ GOOD - Use env vars
CMD ["node", "app.js", "--db=${DATABASE_HOST}"]
```

## **⚡ PRO TIPS**

### **Image Size Optimization**
```dockerfile
# Clean up in same layer
RUN apt-get update && \
    apt-get install -y build-essential && \
    make && \
    apt-get remove -y build-essential && \  # Remove build tools
    apt-get autoremove -y && \
    rm -rf /var/lib/apt/lists/*              # Clear package cache

# Use Alpine variants when possible
FROM node:18-alpine  # ~50MB vs node:18 ~900MB
```

### **Debugging Layers**
```bash
# See layer sizes
docker history myimage --no-trunc

# Inspect specific layer
docker run -it --entrypoint /bin/sh myimage
```

### **Build Arguments vs Environment**
```dockerfile
ARG BUILD_VERSION=latest   # Build-time only
ENV APP_VERSION=$BUILD_VERSION  # Available at runtime

# Build with arg:
docker build --build-arg BUILD_VERSION=v1.0 -t myapp .
```

## **📋 QUICK REFERENCE CARD**

| What you want | Command | Common mistake |
|---------------|---------|----------------|
| Run command once | `RUN` | Putting it after `CMD` |
| Default command | `CMD` | Using `RUN` instead |
| Fixed command | `ENTRYPOINT` | Confusing with `CMD` |
| Copy files | `COPY` | Using `ADD` unnecessarily |
| Set directory | `WORKDIR` | Using `RUN cd` |
| Switch user | `USER` | Creating files after switching |
| Build-time var | `ARG` | Trying to access at runtime |
| Runtime var | `ENV` | Hardcoding values |

## **🔍 The One That Gets Everyone**

```dockerfile
# This doesn't work!
FROM ubuntu
COPY script.sh /
RUN chmod +x /script.sh
USER nobody
CMD /script.sh  # ❌ nobody can't write logs, can't bind ports <1024

# Debug: Container exits immediately with permission error
# Fix: Run as root or use correct user permissions
```

**Remember:** If a container exits immediately, check:
1. Is the command correct? (use `docker logs`)
2. Does the user have permissions?
3. Is the entrypoint/CMD in correct format?
4. Are you missing an interactive shell? (`-it` flag)
