Here's your **Docker Compose "Watch Out!" Cheat Sheet** - the trip points and must-know concepts specifically for Compose:

## 🔥 **THINGS THAT TRIP EVERYONE**

### 1. **Docker Compose versions matter (but less now)**
```yaml
# OLD (version 2) vs NEW (version 3+)
# version: '2'  # Different syntax, fewer features
version: '3.8'   # Modern, use this

# Actually, recent Docker drops version requirement:
# Just 'services:' works in newer versions
```

### 2. **depends_on ≠ waits for ready**
```yaml
services:
  web:
    build: .
    depends_on:
      - db        # ONLY waits for db container to START, not to be READY
      - redis

  db:
    image: postgres  # web might start before Postgres accepts connections!
```

**Fix - add healthcheck + wait script:**
```yaml
services:
  web:
    build: .
    depends_on:
      db:
        condition: service_healthy  # ✅ Waits for healthcheck
  
  db:
    image: postgres
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "postgres"]
      interval: 5s
      timeout: 5s
      retries: 5
```

### 3. **Environment variable expansion - TWO contexts**
```yaml
# .env file
DATABASE_PASSWORD=secret123
PORT=3000

# docker-compose.yml
services:
  app:
    environment:
      - DB_PASS=${DATABASE_PASSWORD}  # Expands from .env file
      - PORT=8080                      # Literal value
    image: nginx:${NGINX_VERSION:-latest}  # Default if not set
```

**Trips people:** Variables expand at different times!
- `$${VAR}` - Escape (literal `${VAR}`)
- `${VAR:-default}` - Use default if VAR unset
- `${VAR:?error}` - Fail with error if VAR unset

### 4. **Volume mapping - host vs container**
```yaml
volumes:
  # ✅ CORRECT - host:container
  - ./data:/app/data
  
  # ❌ WRONG order
  - /app/data:./data  # This means something else!
  
  # Named volume (persists even after down)
  - postgres_data:/var/lib/postgresql/data
  
  # Read-only mount
  - ./config:/app/config:ro
```

### 5. **Port mapping syntax gotchas**
```yaml
ports:
  # Host:Container
  - "3000:3000"      # ✅ Common - map 3000 to 3000
  - "8080:80"        # ✅ Host 8080 → Container 80
  - "80"             # ⚠️ Random host port assigned (docker ps to find it)
  - "127.0.0.1:8080:80"  # ✅ Bind only to localhost
  - "0.0.0.0:8080:80"    # ✅ Bind to all interfaces
```

## 🎯 **MUST-KNOW CONCEPTS**

### **Networking - containers communicate automatically**
```yaml
services:
  web:
    networks:
      - mynetwork
    # Can reach 'db' by service name: http://db:5432
  
  db:
    networks:
      - mynetwork
    # Can reach 'web' as well

networks:
  mynetwork:  # Creates isolated network
```

**Default network exists** - containers in same compose file can reach each other using service names WITHOUT defining networks.

### **Build context vs Dockerfile location**
```yaml
services:
  app1:
    build: .                    # Uses ./Dockerfile
  
  app2:
    build:
      context: ./frontend       # Look in ./frontend folder
      dockerfile: Dockerfile.dev  # Use different filename
  
  app3:
    build:
      context: .
      dockerfile: ./docker/Dockerfile
      args:                      # Build arguments
        NODE_VERSION: 18
```

### **Restart policies - what actually happens**
```yaml
restart: "no"       # Never restart (default)
restart: always     # Always restart, even if stopped manually
restart: on-failure # Only if exit code != 0
restart: unless-stopped  # Restart unless explicitly stopped

# Trips people: 'always' will restart even after 'docker stop'
# You need 'docker compose down' to stop it
```

## 🚨 **COMMON MISTAKES**

### 1. **Not handling config file changes**
```bash
# Changed docker-compose.yml? Need to recreate containers:
docker-compose up -d --force-recreate

# Changed environment variables? Same thing:
docker-compose up -d --force-recreate
```

### 2. **Forgetting .env file precedence**
```yaml
# Precedence (highest to lowest):
# 1. Shell environment variables
# 2. .env file
# 3. Compose file 'environment' section
# 4. Compose file 'env_file' section

# Trips people: Shell vars override .env!
export DATABASE_URL=localhost
# .env has DATABASE_URL=postgres://db - gets overridden!
```

### 3. **Volumes wiping container files**
```yaml
volumes:
  - ./empty_folder:/app/data  # ❌ WIPES OUT /app/data contents!
  
# Container had important files in /app/data? They're gone.
# Empty host folder replaces container folder entirely.
```

### 4. **Command vs entrypoint confusion**
```yaml
services:
  app1:
    image: node
    command: npm start  # Overrides Dockerfile CMD
  
  app2:
    image: node
    entrypoint: /custom-entrypoint.sh
    command: --production  # Args to entrypoint
  
  # Both can be arrays or strings:
  command: ["npm", "start"]  # Exec form (preferred)
  command: npm start         # Shell form
```

### 5. **Container naming collisions**
```yaml
services:
  app:
    container_name: myapp  # Fixed name - can only run ONE!
    
# Without container_name, Compose adds project name:
# myproject_app_1 (can run multiple copies with --scale)
```

## ⚡ **PRO TIPS**

### **Profiles - run groups of services**
```yaml
services:
  app:
    # Always runs
    profiles: []
  
  db:
    # Runs by default
  
  adminer:
    profiles: ["debug", "tools"]  # Only with --profile debug
  
  cadvisor:
    profiles: ["monitoring"]       # Only with --profile monitoring

# Start: docker-compose --profile monitoring up
# Multiple: docker-compose --profile debug --profile monitoring up
```

### **Extending configuration**
```yaml
# docker-compose.base.yml
services:
  app-base:
    environment:
      - LOG_LEVEL=info

# docker-compose.yml
services:
  web:
    extends:
      file: docker-compose.base.yml
      service: app-base
    build: .
```

### **Resource limits - prevent container chaos**
```yaml
services:
  app:
    deploy:
      resources:
        limits:
          cpus: '0.5'      # Half a CPU core
          memory: 512M
        reservations:
          cpus: '0.25'
          memory: 256M
```

## 📋 **QUICK REFERENCE CARD**

| What you want | Command | Common mistake |
|---------------|---------|----------------|
| Rebuild & start | `docker-compose up --build` | Using `docker-compose restart` (won't rebuild) |
| Stop & remove | `docker-compose down` | `docker-compose stop` (leaves containers) |
| Remove volumes too | `docker-compose down -v` | Forgetting `-v`, data persists |
| View logs | `docker-compose logs -f service` | Not specifying service, gets overwhelmed |
| Run one-off command | `docker-compose run app npm test` | Using `exec` (needs running container) |
| Scale service | `docker-compose up --scale web=3` | Not having ports configured dynamically |

## 🔍 **THE ONE THAT GETS EVERYONE**

```yaml
# THIS DOESN'T WORK AS EXPECTED:
services:
  web:
    build: .
    ports:
      - "80:80"
    environment:
      - DATABASE_URL=postgres://db:5432
    depends_on:
      - db
    volumes:
      - .:/app  # Wipes /app/node_modules on host?!
  db:
    image: postgres
    volumes:
      - ./data:/var/lib/postgresql/data  # Will corrupt DB!

# Problems:
# 1. ./data folder must be initialized by Postgres first
# 2. Bind mount overwrites node_modules (use named volume or exclude)
# 3. depends_on doesn't wait for DB to be ready
# 4. Need healthcheck or wait script
```

**Debug checklist when containers won't start:**
1. `docker-compose logs` - What's the actual error?
2. `docker-compose config` - Validates YAML and shows interpolated values
3. `docker-compose down -v && docker-compose up` - Fresh start
4. Check if ports are already in use: `netstat -tulpn | grep :80`
5. Environment variables: `docker-compose run app env | grep DATABASE`

**Remember:** Docker Compose is for DEVELOPMENT and simple deployments. For production, consider Kubernetes, Docker Swarm, or cloud services!
