Diving into Python backends requires understanding two very different philosophies. Currently, the landscape is dominated by Django (the seasoned, heavy-lifting monolith) and FastAPI (the modern, blazing-fast micro-framework).

Here are your extensive notes detailing the core architectural patterns, philosophies, and interview traps for both frameworks.

---

## 1. Django: The "Batteries Included" Monolith

Django's philosophy is "Don't Repeat Yourself" (DRY) and rapid development. It provides everything out of the box: ==an ORM, authentication, an admin panel, and routing.== You do things the "Django Way," or you will fight the framework.

### Core Django Patterns

#### Pattern 1: "Fat Models, Thin Views"

This is the golden rule of Django architecture.

- The Concept: Business logic should live in the Model (or a service layer closely tied to it), not in the View (the controller). Views should only handle HTTP request/response parsing and routing. [[1.1 Django Fat Models, Thin Views]]
    
- Why? Models are highly reusable. If you put logic in a view, you can't easily reuse that logic in a management command, a background task (Celery), or the Django Admin.
    
- Example: Instead of calculating a user's subscription status inside the view, create a `@property` or a method on the `User` model called `is_active_subscriber()`.
    

#### Pattern 2: The Active Record (ish) ORM & Lazy Evaluation

Django's ORM maps Python classes to database tables. It uses a variation of the Active Record pattern.

- Lazy Evaluation: QuerySets are lazy. `users = User.objects.filter(active=True)` does not hit the database. The database is only hit when the QuerySet is evaluated (e.g., iterating over it, printing it, converting it to a list).
    
- The Interview Trap (N+1 Problem): If you loop over a QuerySet and access a related foreign key, Django will make a separate DB query for _each_ iteration.
    
    - _Fix:_ ==Use `select_related()` (for ForeignKey/OneToOne - does a SQL JOIN) or `prefetch_related()` (for ManyToMany/Reverse FK - does a separate query and joins in Python).==
        

#### Pattern 3: Class-Based Views (CBVs) & Mixins (Django REST Framework)

While Function-Based Views (FBVs) are simpler, modern Django (especially with Django REST Framework - DRF) heavily utilizes CBVs.

- The Concept: Using Object-Oriented Programming to handle HTTP methods (`get`, `post`, `put`) as class methods.
    
- Mixins: DRF relies on Mixins (multiple inheritance) to compose behavior. You can build an API endpoint by inheriting `ListModelMixin`, `CreateModelMixin`, and `GenericAPIView`.
    
- _Pros:_ Massive code reduction for standard CRUD operations.
    
- _Cons:_ Can lead to "Inheritance Hell" where the control flow becomes difficult to trace.
    

#### Pattern 4: Signals (The Observer Pattern)

Signals allow decoupled applications to get notified when actions occur elsewhere in the framework.

- Use Case: Automatically creating a `UserProfile` when a `User` is created (`post_save` signal).
    
- The Warning: Signals are synchronous. They block the main thread. They also hide control flow, making debugging incredibly difficult. Best practice: Avoid them unless absolutely necessary. Use explicit function calls or background tasks instead.
    

#### Pattern 5: Middleware

A chain of hooks into Django’s request/response processing. It’s a lightweight plugin system for globally altering input or output.

- Use Cases: Authentication verification, CORS handling, adding request IDs for logging, or checking rate limits before the request ever reaches the View.

## 2. FastAPI: The Modern Async Micro-Framework

FastAPI's philosophy is explicit validation, high performance, and asynchronous processing. It does not include an ORM, an admin panel, or even a database connection out of the box. You assemble the pieces yourself.

### Core FastAPI Patterns

#### Pattern 1: Dependency Injection (DI)

This is the beating heart of FastAPI.

- The Concept: Instead of a function creating its own dependencies (like a database connection or a user session), the framework _injects_ them into the function as arguments.
    
- Why? It makes testing incredibly easy. You can swap out a real database dependency for a mock one during testing using `app.dependency_overrides`.
    
- Example:
    
    Python
    
    ```
    @app.get("/users/")
    def get_users(db: Session = Depends(get_db)): 
        # get_db is a dependency that yields a DB session
        return db.query(User).all()
    ```
    

#### Pattern 2: Pydantic DTOs (Data Transfer Objects)

FastAPI uses Pydantic to handle data validation and serialization. Pydantic enforces type hints at runtime.

- The Concept: You define data schemas using standard Python type hints.
    
- Use Case: You separate your Database Models (e.g., SQLAlchemy classes) from your API Models (Pydantic classes). This ensures you never accidentally leak sensitive DB fields (like a password hash) to the API response. You use a `UserCreate` Pydantic model for input, and a `UserResponse` Pydantic model for output.
    

#### Pattern 3: Asynchronous Programming (ASGI vs WSGI)

Django is traditionally WSGI (Web Server Gateway Interface) (synchronous). FastAPI is built on ASGI (Asynchronous Server Gateway Interface) via Starlette.

- The Concept: Using `async def` and `await`. When an API calls a slow database or a third-party service, an `async` framework releases the thread back to the event loop to handle other requests while waiting.
    
- The Interview Trap (Blocking the Event Loop): If you use `async def` for a route, but put a blocking, synchronous operation inside it (like `time.sleep()` or a heavy CPU-bound calculation), you will freeze the entire server.
    
    - _Fix:_ Run CPU-bound tasks in a background thread/process, or simply define the route as a standard `def` (FastAPI is smart enough to run standard `def` routes in an external threadpool).
        

#### Pattern 4: The Repository Pattern

Because FastAPI is agnostic about databases, developers often implement the Repository Pattern to abstract data access.

- The Concept: The API router never imports SQLAlchemy or raw SQL. Instead, it calls a "Repository" class (`UserRepository.get_by_id()`).
    
- Why? If you ever need to switch from PostgreSQL to MongoDB, you only rewrite the Repository layer; your API endpoints remain untouched.

## 3. Architectural Showdown: When to use Which?

| Feature       | Django (with DRF)                                                        | FastAPI                                              |
| ------------- | ------------------------------------------------------------------------ | ---------------------------------------------------- |
| Architecture  | MVC (Model/View/Controller) / Monolithic                                 | Micro-framework / Modular                            |
| Pacing        | "Batteries included" (Fast setup for CRUD apps)                          | "Build it yourself" (Slower initial setup, explicit) |
| Performance   | Good. I/O bound tasks can bottleneck.                                    | Exceptional. Handles concurrent I/O flawlessly.      |
| Validation    | DRF Serializers (Functional, but clunky)                                 | Pydantic (Type-hint driven, standard Python)         |
| Documentation | Requires third-party packages (drf-yasg) (yet-another-swagger-generator) | Automatic via OpenAPI (Swagger UI out of the box)    |

- **Choose Django** if: You are building a standard CRUD application, an e-commerce site, a CMS, or a SaaS where time-to-market is critical and you need an Admin panel, Auth, and DB migrations (Alembic in FastAPI is great, but Django's built-in migrations are unmatched) immediately.
    
- **Choose FastAPI** if: You are building a high-performance microservice, heavily relying on WebSockets, machine learning model serving (FastAPI is the standard for MLOps), or an API that orchestrates many third-party integrations (highly I/O bound).

## 4. Universal Python Backend Best Practices

Regardless of the framework, mature Python backends utilize these patterns:

- **Background Tasks (Message Queues)**: Never send emails, process images, or generate heavy reports in the HTTP request cycle. Offload them. Use Celery with Redis/RabbitMQ, or RQ (Redis Queue).
    
- **12-Factor App Configuration:** Never hardcode secrets. Use environment variables. In FastAPI, this is managed natively using `pydantic-settings`. In Django, developers typically use `django-environ`.
    
- **Database Migrations**: Version control your database schema. In Django, use `manage.py makemigrations`. In FastAPI (with SQLAlchemy), use Alembic.
