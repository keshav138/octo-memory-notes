## 2. Django — Core Concepts

### Project Structure

```
myproject/
├── manage.py
├── myproject/
│   ├── settings.py       ← config (DB, apps, middleware)
│   ├── urls.py           ← root URL router
│   ├── wsgi.py           ← WSGI entry point (sync)
│   └── asgi.py           ← ASGI entry point (async)
└── myapp/
    ├── models.py         ← DB schema (ORM)
    ├── views.py          ← request handlers
    ├── urls.py           ← app-level routes
    ├── serializers.py    ← data validation + transformation (DRF)
    ├── admin.py          ← admin panel config
    └── migrations/       ← DB migration files
```

### Request Lifecycle in Django

```
HTTP Request
    ↓
WSGI/ASGI Server (Gunicorn/Uvicorn)
    ↓
Django Middleware Stack (top → down)
    ↓
URL Router (urls.py) → matches pattern
    ↓
View (views.py) → business logic
    ↓
ORM → Database query
    ↓
Response → Middleware Stack (bottom → up)
    ↓
HTTP Response
```

## 3. Django ORM

The ORM maps Python classes → DB tables. No raw SQL needed (though you can use it).

### Defining Models

```python
from django.db import models

class User(models.Model):
    name = models.CharField(max_length=100)
    email = models.EmailField(unique=True)
    created_at = models.DateTimeField(auto_now_add=True)
    is_active = models.BooleanField(default=True)

    class Meta:
        db_table = "users"          # custom table name
        ordering = ["-created_at"]  # default sort

class Order(models.Model):
    user = models.ForeignKey(User, on_delete=models.CASCADE, related_name="orders")
    total = models.DecimalField(max_digits=10, decimal_places=2)
    status = models.CharField(max_length=20, choices=[
        ("pending", "Pending"),
        ("paid", "Paid"),
        ("cancelled", "Cancelled"),
    ])
```

### Common ORM Queries

```python
# CREATE
user = User.objects.create(name="Keshav", email="k@test.com")

# READ
User.objects.all()                          # all users
User.objects.get(id=1)                      # single, raises exception if not found
User.objects.filter(is_active=True)         # queryset
User.objects.filter(name__icontains="kes")  # case-insensitive contains
User.objects.exclude(is_active=False)       # exclude
User.objects.order_by("-created_at")[:10]   # latest 10

# UPDATE
User.objects.filter(id=1).update(name="New Name")

# DELETE
User.objects.filter(is_active=False).delete()

# AGGREGATE
from django.db.models import Count, Avg, Sum
User.objects.aggregate(total=Count("id"))
Order.objects.filter(user=user).aggregate(total=Sum("total"))

# RELATED (FK traversal)
user.orders.all()                           # all orders for a user
Order.objects.select_related("user")        # JOIN — use for ForeignKey (1-to-1, M-to-1)
Order.objects.prefetch_related("items")     # separate query — use for M2M or reverse FK
```

### N+1 Problem — Critical Interview Topic

```python
# ❌ BAD — N+1 queries (1 for orders + 1 per order for user)
orders = Order.objects.all()
for order in orders:
    print(order.user.name)  # hits DB every iteration

# ✅ GOOD — select_related does a JOIN, single query
orders = Order.objects.select_related("user").all()
for order in orders:
    print(order.user.name)  # no extra DB hit
```

> **N+1 problem:** Fetching a list of N items, then making 1 DB call per item for related data = N+1 total queries. Fix with `select_related` (JOIN) or `prefetch_related` (batched second query).

## 4. Django REST Framework (DRF)

DRF adds serializers, ViewSets, and routers on top of Django for building APIs fast.

### Serializers

Serializers convert complex types (QuerySets, model instances) ↔ Python dicts ↔ JSON.

```python
from rest_framework import serializers
from .models import User

class UserSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ["id", "name", "email", "created_at"]
        read_only_fields = ["id", "created_at"]
        extra_kwargs = {
            "email": {"required": True}
        }

    def validate_email(self, value):
        # Custom field-level validation
        if "spam" in value:
            raise serializers.ValidationError("No spam emails.")
        return value

    def validate(self, data):
        # Object-level validation (cross-field)
        return data
```

### Views — 3 Ways

```python
# 1. Function-Based View (FBV) — explicit, verbose
from rest_framework.decorators import api_view
from rest_framework.response import Response

@api_view(["GET", "POST"])
def user_list(request):
    if request.method == "GET":
        users = User.objects.all()
        return Response(UserSerializer(users, many=True).data)
    elif request.method == "POST":
        serializer = UserSerializer(data=request.data)
        if serializer.is_valid():
            serializer.save()
            return Response(serializer.data, status=201)
        return Response(serializer.errors, status=400)


# 2. Class-Based View (CBV) — more structured
from rest_framework.views import APIView

class UserListView(APIView):
    def get(self, request):
        users = User.objects.all()
        return Response(UserSerializer(users, many=True).data)

    def post(self, request):
        serializer = UserSerializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        serializer.save()
        return Response(serializer.data, status=201)


# 3. ViewSet + Router — least code, most magic
from rest_framework.viewsets import ModelViewSet
from rest_framework.routers import DefaultRouter

class UserViewSet(ModelViewSet):
    queryset = User.objects.all()
    serializer_class = UserSerializer

# In urls.py:
router = DefaultRouter()
router.register("users", UserViewSet)
urlpatterns = router.urls
# Auto-generates: GET/POST /users/, GET/PUT/PATCH/DELETE /users/{id}/
```

### Permissions & Authentication in DRF

```python
from rest_framework.permissions import IsAuthenticated, IsAdminUser
from rest_framework_simplejwt.authentication import JWTAuthentication

class UserViewSet(ModelViewSet):
    authentication_classes = [JWTAuthentication]
    permission_classes = [IsAuthenticated]

    def get_permissions(self):
        # Custom per-action permissions
        if self.action == "destroy":
            return [IsAdminUser()]
        return [IsAuthenticated()]
```

## 5. FastAPI — Core Concepts

FastAPI is built on Starlette (ASGI) and Pydantic. It's async-first and auto-generates OpenAPI docs.

### Basic App

```python
from fastapi import FastAPI, HTTPException, Depends
from pydantic import BaseModel, EmailStr
from typing import Optional

app = FastAPI()

class UserCreate(BaseModel):
    name: str
    email: EmailStr
    age: Optional[int] = None

class UserResponse(BaseModel):
    id: int
    name: str
    email: str

    class Config:
        from_attributes = True  # allows ORM model → Pydantic

@app.get("/users", response_model=list[UserResponse])
async def get_users():
    return await fetch_users_from_db()

@app.post("/users", response_model=UserResponse, status_code=201)
async def create_user(user: UserCreate):
    if await email_exists(user.email):
        raise HTTPException(status_code=409, detail="Email already exists")
    return await save_user(user)
```

### Path, Query, Body Parameters

```python
@app.get("/users/{user_id}")          # path param
async def get_user(user_id: int):     # FastAPI validates type automatically
    ...

@app.get("/products")
async def list_products(
    category: str,                    # required query param
    page: int = 1,                    # optional query param with default
    limit: int = 20,
):
    ...
```

### Dependency Injection

FastAPI's `Depends()` is how you share logic — DB sessions, auth, config.

```python
from fastapi import Depends
from sqlalchemy.orm import Session

def get_db():
    db = SessionLocal()
    try:
        yield db           # yield = runs cleanup after request
    finally:
        db.close()

def get_current_user(token: str = Depends(oauth2_scheme), db: Session = Depends(get_db)):
    payload = decode_jwt(token)
    user = db.query(User).get(payload["user_id"])
    if not user:
        raise HTTPException(status_code=401)
    return user

@app.get("/me")
async def get_me(current_user: User = Depends(get_current_user)):
    return current_user
```

### Pydantic Validation

```python
from pydantic import BaseModel, validator, Field

class OrderCreate(BaseModel):
    product_id: int
    quantity: int = Field(..., gt=0, le=1000)  # must be 1–1000
    notes: Optional[str] = Field(None, max_length=500)

    @validator("quantity")
    def quantity_must_be_positive(cls, v):
        if v <= 0:
            raise ValueError("Quantity must be positive")
        return v
```

Unlike Django, which is a "batteries-included" framework with its own built-in ORM (Object-Relational Mapper), **FastAPI does not have a built-in database layer.** FastAPI is strictly a web framework. It handles the **HTTP layer** (requests/responses), and it leaves the **Database layer** entirely up to you. However, it provides a powerful tool called **Dependency Injection** that makes connecting to any database feel seamless.

### 1. The "Middlemen": ORMs and ODMs

Because FastAPI doesn't dictate how you talk to your database, you choose a library to act as the bridge. These libraries do the "Django-style" work:

- **For SQL (PostgreSQL, MySQL, SQLite):** You use **SQLAlchemy** or **SQLModel**.
    
- **For NoSQL (MongoDB):** You use **Beanie** or **Motor**.
    
- **For NoSQL (Redis):** You use **Redis-py**.
    

### 2. How the Data Flows

In Django, you use `Model.objects.save()`. In FastAPI, the flow is broken down into three distinct steps to ensure maximum speed and type safety:

1. **The Request (Pydantic):** FastAPI receives JSON. It uses a **Pydantic Schema** to validate that the data is correct (e.g., "Is this an email?", "Is this age a number?").
    
2. **The Dependency (The Session):** FastAPI uses "Dependency Injection" (`Depends`) to open a connection to the database only when a request hits a specific route.
    
3. **The Transfer (The ORM):** You take that validated Pydantic data, convert it into a **Database Model** (like SQLAlchemy), and tell the ORM to save it.
    

### 3. Comparison: Django vs. FastAPI

|**Feature**|**Django**|**FastAPI**|
|---|---|---|
|**Database Engine**|Built-in Django ORM|Choice of SQLAlchemy, SQLModel, Tortoise, etc.|
|**Migrations**|Built-in (`makemigrations`)|External (usually **Alembic**)|
|**Validation**|Django Forms / Serializers|**Pydantic** (Native Python type hints)|
|**Async Support**|Partial / Improving|**Native & Core** (Built for high concurrency)|
|**Flexibility**|High (but follows "The Django Way")|Extreme (plug-and-play any library)|

### 4. The "Modern Way": SQLModel

Since you mentioned the "typing" aspect earlier, **SQLModel** is the closest thing to the Django experience in the FastAPI ecosystem. It was designed by the creator of FastAPI to bridge the gap between your database and your code.

In Django, your model is both the database table and the data validator. SQLModel does the same thing:

Python

```
from sqlmodel import SQLModel, Field

# This acts as BOTH the DB table and the Pydantic schema
class Hero(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str
    secret_name: str
    age: int | None = None
```

### Summary

While Django provides a "black box" that handles the DB for you, FastAPI requires you to **plug in a library** (like SQLAlchemy) and **inject a session** into your routes. It’s slightly more setup, but it results in code that is significantly faster and easier to debug because every data transfer is explicitly typed.

## 6. Async Python — Critical for FastAPI

### sync vs async

```python
import asyncio
import httpx

# ❌ Blocking — holds up the entire event loop
def fetch_data():
    import requests
    return requests.get("https://api.example.com/data").json()

# ✅ Non-blocking — yields control while waiting
async def fetch_data():
    async with httpx.AsyncClient() as client:
        response = await client.get("https://api.example.com/data")
        return response.json()
```

### async / await rules

```python
async def main():
    # await = pause here, let event loop do other things
    result = await fetch_data()

    # Run multiple async tasks concurrently
    results = await asyncio.gather(
        fetch_user(1),
        fetch_orders(1),
        fetch_profile(1),
    )
    # All 3 run in parallel, not sequentially

asyncio.run(main())
```

### When to use async?

|Scenario|Use|
|---|---|
|DB queries (heavy computation)|sync is fine (use threadpool)|
|HTTP calls to external APIs|`async` + `await`|
|File I/O|`async` with `aiofiles`|
|Redis/cache calls|`async` with `aioredis`|
|CPU-heavy work (ML inference)|`multiprocessing` or `BackgroundTasks`|

> **Key insight:** async helps with I/O-bound tasks. For CPU-bound tasks, use separate workers/processes — the GIL prevents true parallel threads in Python.

## 7. Middleware

Middleware wraps every request/response. Used for logging, auth, CORS, rate limiting.

### Django Middleware

```python
# settings.py
MIDDLEWARE = [
    "django.middleware.security.SecurityMiddleware",
    "django.contrib.sessions.middleware.SessionMiddleware",
    "corsheaders.middleware.CorsMiddleware",
    "django.middleware.common.CommonMiddleware",
    "django.contrib.auth.middleware.AuthenticationMiddleware",
    "myapp.middleware.RequestLoggingMiddleware",   # custom
]

# Custom middleware
class RequestLoggingMiddleware:
    def __init__(self, get_response):
        self.get_response = get_response

    def __call__(self, request):
        print(f"→ {request.method} {request.path}")
        response = self.get_response(request)  # call next middleware/view
        print(f"← {response.status_code}")
        return response
```

### FastAPI Middleware

```python
from fastapi.middleware.cors import CORSMiddleware
from starlette.middleware.base import BaseHTTPMiddleware
import time

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://app.com"],
    allow_methods=["*"],
    allow_headers=["*"],
)

class TimingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.time()
        response = await call_next(request)
        duration = time.time() - start
        response.headers["X-Process-Time"] = str(duration)
        return response

app.add_middleware(TimingMiddleware)
```

## 8. Background Tasks & Celery

Long-running tasks (sending emails, processing files, ML inference) should NOT block the HTTP request.

[[4. OS Performance - Threading, Processing]]
### FastAPI BackgroundTasks (lightweight)

```python
from fastapi import BackgroundTasks

def send_welcome_email(email: str):
    # runs after response is sent
    email_client.send(email, "Welcome!")

@app.post("/users")
async def create_user(user: UserCreate, background_tasks: BackgroundTasks):
    new_user = await save_user(user)
    background_tasks.add_task(send_welcome_email, user.email)
    return new_user  # response sent immediately
```

### Celery (heavy async task queue)

```python
# celery.py
from celery import Celery

app = Celery("myapp", broker="redis://localhost:6379/0")

@app.task
def process_order(order_id: int):
    order = Order.objects.get(id=order_id)
    # heavy processing...
    order.status = "processed"
    order.save()

# In view:
process_order.delay(order_id=42)   # sends to Redis queue, returns immediately
```

**Celery components:**

- **Broker** (Redis/RabbitMQ): Message queue
- **Worker**: Process that picks up and runs tasks
- **Beat**: Scheduler for periodic tasks (cron-like)
- **Flower**: Web UI to monitor tasks

## 9. Caching with Redis

```python
import redis
import json

r = redis.Redis(host="localhost", port=6379, db=0)

def get_user(user_id: int):
    cache_key = f"user:{user_id}"

    # Try cache first
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)

    # Cache miss — hit DB
    user = User.objects.get(id=user_id)
    data = UserSerializer(user).data

    # Store in cache with 5 min TTL
    r.setex(cache_key, 300, json.dumps(data))
    return data
```

**Cache invalidation strategies:**

- **TTL (Time To Live)**: Expire after N seconds — simple, eventual consistency
- **Write-through**: Update cache on every DB write — always fresh, more writes
- **Cache-aside** (above pattern): App manages cache manually — most common
- **Invalidate on write**: Delete cache key when resource is updated

## 12. Quick-Fire Interview Q&A

**Q: What is the N+1 problem and how do you fix it?**  
A: When fetching N objects triggers N additional queries for related data. Fix with `select_related` (JOIN) for ForeignKey or `prefetch_related` (batch query) for M2M/reverse FK.

**Q: What's the difference between WSGI and ASGI?**  
A: WSGI is synchronous — one request per thread. ASGI is async — can handle many concurrent requests in a single thread via the event loop. FastAPI uses ASGI; traditional Django uses WSGI (but supports ASGI via channels).

**Q: How do you handle a slow task like sending an email after signup?**  
A: Background task — either FastAPI's `BackgroundTasks` for lightweight jobs or Celery + Redis for distributed, retryable task queues.

**Q: What is Pydantic and why use it?**  
A: Data validation library. Defines data shapes as Python classes with type hints. Automatically validates and converts incoming data — raises clear errors on invalid input. FastAPI uses it for request/response models.

**Q: How does Django's middleware work?**  
A: It's a stack of hooks that wraps every request and response. Each middleware processes the request going in and the response coming out. Used for auth, logging, CORS, security headers, etc.

**Q: How do you prevent your Django app from leaking secrets?**  
A: Environment variables + `.env` file (never committed to Git). Use `python-decouple` or `pydantic-settings`. Add `.env` to `.gitignore`.

**Q: What's the GIL and why does it matter for backend Python?**  
A: Global Interpreter Lock — only one Python thread executes at a time. This means CPU-bound tasks don't benefit from threading. For I/O-bound work (DB, HTTP) async/threads still help because threads release the GIL while waiting. For CPU-heavy work, use `multiprocessing` or separate worker processes (Celery workers).
