Understanding how to secure your applications is just as important as knowing how to build them. Before diving into the specific methods listed in your image, it is crucial to establish the golden rule of security: **Authentication (AuthN)** is proving _who you are_ (verifying identity), while **Authorization (AuthZ)** is proving _what you are allowed to do_ (verifying permissions).

Here are your extensive, interview-ready notes on the five core authentication and authorization methods.

---

## **1. Basic Authentication**

Basic Auth is the oldest, simplest, and most fundamental way to protect an HTTP endpoint.

- **How it Works:** 1. The client takes their username and password and joins them with a colon (e.g., `admin:password123`).
    
    2. The client encodes this string using Base64.
    
    3. The client sends this encoded string in the HTTP header of every single request: `Authorization: Basic YWRtaW46cGFzc3dvcmQxMjM=`.
    
    4. The server decodes it, checks the database, and grants access.
    
- **The Architecture Trap:** Base64 is **encoding, not encryption**. Anyone intercepting the network traffic can decode it instantly.
    
- **Pros:** Exceptionally easy to implement. Supported natively by all browsers and HTTP clients.
    
- **Cons:** Extremely insecure over plain HTTP. It forces the server to verify the password hash against the database on _every single request_, which is computationally expensive.
    
- **When to use:** Internal tooling, quick prototypes, or legacy systems, and **strictly over HTTPS**.
    

---

## **2. API Keys**

An API key is a long, randomly generated, unique string provided by the server to the client.

- **How it Works:**
    
    1. A developer registers their application on your platform (e.g., Stripe, Google Maps).
        
    2. You generate a unique string (the API Key) and give it to them.
        
    3. The developer includes this key in the header (`x-api-key: 12345-ABCDE`) or query string (`?api_key=12345-ABCDE`) of their requests.
        
    4. The server looks up the key in a database to identify the caller and check their rate limits.
        
- **Pros:** Simple to implement, easy to track usage (billing/rate limiting), and great for machine-to-machine (M2M) communication.
    
- **Cons:** It identifies the _calling application_, not the _individual user_ using the application. If the key is hardcoded into a frontend app (like a mobile app or React site), anyone can extract it and abuse it.
    
- **When to use:** Public APIs providing data (weather, maps), or server-to-server microservice communication.
    

---

## **3. Session + Cookie (Stateful Authentication)**

This is the traditional way the web has handled user logins for decades. It relies on the server remembering who is logged in (maintaining state).

- **How it Works:**
    
    1. The user submits their username and password via a login form.
        
    2. The server verifies the credentials. If correct, the server generates a unique, random `session_id` and stores it in its own memory or a database (like Redis).
        
    3. The server sends this `session_id` back to the browser via a `Set-Cookie` HTTP header.
        
    4. **The Magic:** The browser automatically stores this cookie and attaches it to _every subsequent request_ to that domain.
        
    5. The server reads the cookie, looks up the `session_id` in its database, and knows who the user is.
        
- **Pros:** Highly secure. You can make cookies `HttpOnly` (JavaScript cannot read them, preventing XSS attacks) and `Secure` (only sent over HTTPS). It is also trivial to revoke access—the server just deletes the session from its database.
    
- **Cons:** **Scalability.** Because it is stateful, if you have 10 load-balanced servers, and User A logs into Server 1, Server 2 doesn't know who they are unless you implement a centralized session store (like Redis) that all 10 servers share.
    
- **When to use:** Traditional server-rendered web applications (Django, Ruby on Rails) where the frontend and backend live on the same domain.
    

---

## **4. JWT (JSON Web Tokens) (Stateless Authentication)**

JWTs were popularized to solve the horizontal scaling issues of Session/Cookie architecture.

- **How it Works:**
    
    1. The user logs in with credentials.
        
    2. The server verifies them and creates a JSON object containing user data (e.g., `{"user_id": 123, "role": "admin"}`).
        
    3. The server cryptographically **signs** this JSON object using a secret key only the server knows, creating a JWT.
        
    4. The server sends the JWT to the client. The client stores it and sends it back in the header: `Authorization: Bearer <token>`.
        
    5. **The Magic:** When the server receives the token, it doesn't need to check a database. It simply recalculates the cryptographic signature. If the signature matches, the server knows the token is valid and hasn't been tampered with. [[2.3.1 JWT Stateless Example]]
        
- **Pros:** Completely stateless. You can scale to 10,000 servers without a centralized Redis cache. Great for mobile apps and single-page applications (React/Vue).
    
- **Cons:** **Revocation is a nightmare.** Because the server doesn't keep track of active tokens, you cannot easily "log a user out" or ban them instantly. The token remains valid until its built-in expiration time runs out. (To fix this, companies build "blacklists", which defeats the purpose of being stateless).
    
- **When to use:** Modern REST APIs, microservices architectures, and mobile backend APIs.

## **5. OAuth 2.0**

OAuth 2.0 is an industry-standard **authorization framework**. It is specifically designed to allow a user to grant a third-party application limited access to their resources without handing over their password. (e.g., "Allow Grammarly to read my Google Docs").

- **How it Works (The Authorization Code Flow):**
    
    1. The user clicks "Log in with Google" on your app.
        
    2. Your app redirects the user to Google's actual website.
        
    3. The user logs into Google (your app never sees their password) and clicks "Allow access."
        
    4. Google redirects the user back to your app, passing along a temporary "Authorization Code."
        
    5. Your backend server takes this code and secretly trades it with Google's servers for an **Access Token** (and usually a Refresh Token).
        
    6. Your app uses the Access Token to make requests to Google's API on the user's behalf.
        
- **Pros:** Unparalleled security for delegated access. Users don't have to create new passwords for every site.
    
- **Cons:** Highly complex to implement correctly. There are multiple "flows" (Authorization Code, Implicit, Client Credentials) depending on whether the client is a server, a browser, or a mobile app.
    
- **When to use:** Whenever you need to access data on behalf of a user from another platform (Google, GitHub, Facebook), or if you are building an enterprise Single Sign-On (SSO) system using OpenID Connect (which is built on top of OAuth 2.0).
