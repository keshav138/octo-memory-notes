Mastering REST API design is an excellent goal. It is a massive topic, but breaking it down by principles, mechanics, and best practices makes it highly digestible.

Here are extensive, structured notes designed to equip you to tackle any interview or architectural question regarding REST.

---

## **1. The Core Philosophy of REST**

REST (**Re**presentational **S**tate **T**ransfer) is an architectural style introduced by Roy Fielding in 2000. It is not a protocol or a standard; it is a set of constraints applied to the architecture of distributed systems, most commonly implemented over HTTP.

### **The 6 Guiding Constraints**

For an API to be considered truly RESTful, it must adhere to these constraints:

1. **Client-Server:** The client (frontend) and server (backend) are decoupled. The client handles the UI and user state, while the server handles data storage and backend logic. They evolve independently.
    
2. **Statelessness:** **(Crucial Interview Concept)** The server must not store any session state about the client. _Every_ request from the client must contain all the information necessary for the server to understand and process it (e.g., auth tokens).When we say the server is **stateless**, we mean that for every individual request, the server has everything it needs to fulfill that request **contained within the request itself.**

- **Does a JWT store information?** Yes. It stores user ID, roles, and expiration.
    
- **Does the Server store that JWT?** No. This is the crucial part. The server verifies the token’s signature using a secret key, but it doesn't keep a list of "currently logged-in tokens" in its RAM.


    
3. **Cacheability:** Responses must implicitly or explicitly define themselves as cacheable or non-cacheable. This improves scalability and performance.
    
4. **Layered System:** A client cannot ordinarily tell whether it is connected directly to the end server or to an intermediary (load balancer, cache, proxy) along the way.
    
5. **Uniform Interface:** This simplifies the architecture by decoupling the implementations from the services they provide. It relies on four sub-principles:
    
    - **Resource Identification:** Resources are identified in requests (via URIs).
        
    - **Resource Manipulation via Representations:** A client holds a representation of a resource (like a JSON object) and uses it to modify or delete the resource on the server.
        
    - **Self-Descriptive Messages:** Each message includes enough information to describe how to process it (e.g., `Content-Type: application/json`).
        
    - **HATEOAS (Hypermedia As The Engine Of Application State):** The server sends back links in the response, allowing the client to dynamically navigate the API without hardcoded endpoints. _(Note: Most real-world APIs skip this, resulting in "Pragmatic REST" rather than strictly pure REST)._
        
6. **Code on Demand (Optional):** Servers can temporarily extend or customize the functionality of a client by transferring executable code (e.g., JavaScript).

## **2. Resource Naming & URI Design**

REST is resource-centric, not action-centric. You interact with _nouns_, not _verbs_.

- **Use Plural Nouns:** Use `/users` instead of `/user` or `/getUsers`.
    
- **Keep it Hierarchical:** Represent relationships logically.
    
    - _Good:_ `/users/123/orders/456` (Order 456 belonging to User 123).
        
- **Limit Nesting:** Avoid absurdly long URIs. Cap it at two or three levels.
    
    - _Bad:_ `/users/123/orders/456/items/789/shipping`
        
    - _Good:_ `/orders/456/items` (Once you know the order, you don't need the user ID in the URI).
        
- **Use Hyphens for Readability:** Use `/user-profiles`, not `/user_profiles` or `/userProfiles`.

## **3. HTTP Methods & Idempotency**

Understanding which verb to use and how it affects the server state is a frequent test of REST knowledge.

|**Method**|**CRUD Action**|**Description**|**Idempotent?**|**Safe?**|
|---|---|---|---|---|
|**GET**|Read|Retrieves a resource or collection.|**Yes**|**Yes**|
|**POST**|Create|Creates a new resource.|No|No|
|**PUT**|Update|**Replaces** an entire resource. If it doesn't exist, it may create it.|**Yes**|No|
|**PATCH**|Update|**Partially** updates a resource.|No*|No|
|**DELETE**|Delete|Removes a resource.|**Yes**|No|

- **Safe:** The request does not alter the state of the server (it's read-only).
    
- **Idempotent:** Making the exact same request multiple times has the _same result_ as making it once.
    
    - _Example:_ If you `DELETE /users/123` ten times, the end state is the same: the user is gone. If you `POST /users` ten times, you create ten identical users.
        
    - _*Note on PATCH:_ PATCH is technically not idempotent by default, but it can be implemented to be idempotent depending on the exact payload logic.

## **4. Status Codes**

Never return a `200 OK` with an error message in the payload. Use the HTTP standard correctly.

- **2xx (Success):**
    
    - `200 OK`: Standard success for `GET`, `PUT`, `PATCH`.
        
    - `201 Created`: Standard success for `POST`.
        
    - `204 No Content`: Success, but nothing to return (common for `DELETE`).
        
- **3xx (Redirection):**
    
    - `301 Moved Permanently`: The URI has changed.
        
    - `304 Not Modified`: The client's cached version is still valid.
        
- **4xx (Client Errors - The client messed up):**
    
    - `400 Bad Request`: Malformed syntax or validation error.
        
    - `401 Unauthorized`: Authentication is missing or invalid (Who are you?).
        
    - `403 Forbidden`: Authenticated, but lacks permissions (I know who you are, but you can't do this).
        
    - `404 Not Found`: Resource does not exist.
        
    - `405 Method Not Allowed`: The resource exists, but the HTTP verb used is not supported.
        
    - `429 Too Many Requests`: Rate limiting triggered.
        
- **5xx (Server Errors - The server messed up):**
    
    - `500 Internal Server Error`: Generic crash/unhandled exception.
        
    - `503 Service Unavailable`: Server is down or overloaded.

## **5. API Authentication & Authorization**

Because REST is stateless, every request must prove who is making it.

### **Common Strategies:**

1. **Basic Authentication:** * Sends a Base64 encoded `username:password` string in the header (`Authorization: Basic base64_string`).
    
    - _Pros:_ Extremely simple.
        
    - _Cons:_ Highly insecure unless strictly over HTTPS. Rare in modern APIs.
        
2. **API Keys:**
    
    - A unique string generated by the server, passed via headers (`x-api-key: 12345`) or query params.
        
    - _Pros:_ Easy to implement, good for server-to-server communication.
        
    - _Cons:_ Hard to revoke securely if leaked; doesn't identify the _end user_, just the calling application.
        
3. **JSON Web Tokens (JWT) / Bearer Tokens:**
    
    - A signed, base64-encoded JSON object passed in the header (`Authorization: Bearer <token>`).
        
    - _Pros:_ Completely stateless. The server doesn't need to query a database to verify the user; it just validates the cryptographic signature of the token.
        
    - _Cons:_ Cannot be easily revoked before expiration without implementing a stateful "blacklist".
        
4. **OAuth 2.0 / OpenID Connect:**
    
    - An authorization framework that allows third-party applications to obtain limited access to an HTTP service (e.g., "Log in with Google").
        
    - Uses short-lived _Access Tokens_ and long-lived _Refresh Tokens_.

## **6. Robust Error Handling**

Good APIs don't just fail; they explain _why_ they failed in a predictable format. Use a standard schema for all errors.

**Best Practices:**

- Always return the correct HTTP status code.
    
- Never expose internal stack traces to the client.
    
- Provide a standardized JSON error body (e.g., RFC 7807 Problem Details).
    

**Example of an excellent error response:**

JSON

```
{
  "status": 400,
  "error": "Bad Request",
  "message": "Validation Failed",
  "code": "VALIDATION_ERROR_001",
  "details": [
    {
      "field": "email",
      "issue": "Invalid format"
    },
    {
      "field": "password",
      "issue": "Must be at least 8 characters"
    }
  ],
  "help_url": "https://api.example.com/docs/errors#VALIDATION_ERROR_001"
}
```

## **7. Advanced Patterns (Interview Differentiators)**

### **Pagination**

Never return huge datasets at once. Use pagination.

- **Offset/Limit (Traditional):** `GET /users?offset=100&limit=20`. Simple, but slow on massive databases because the DB still has to count rows before skipping.
    
- **Cursor-based (Modern):** `GET /users?cursor=eyJwYWdlIjoxfQ==&limit=20`. Passes a pointer to the last viewed record. Much faster for large datasets (e.g., Twitter timeline), but harder to implement "jump to page 5".
    

### **Filtering, Sorting, and Searching**

Keep the URI clean and push complexities to the query string.

- **Filtering:** `GET /users?role=admin&status=active`
    
- **Sorting:** `GET /users?sort=-created_at,name` (The `-` denotes descending order).
    
- **Searching:** `GET /users?q=John+Doe`
    

### **Versioning**

APIs evolve. You must not break existing clients.

- **URI Versioning (Most Common):** `GET /v1/users` (Easy to route, highly visible).
    
- **Header / Content Negotiation (Most RESTful):** Clients request a specific version via headers: `Accept: application/vnd.myapi.v2+json`. (Cleaner URIs, but harder to test in a simple browser).
    

### **Caching Mechanics**

Implement caching to save bandwidth and compute.

- **ETags (Entity Tags):** The server sends a hash of the resource. Next time, the client asks: "Give me the resource, but only if the ETag has changed from `hash123`" (`If-None-Match: hash123`). If it hasn't changed, the server returns `304 Not Modified` and an empty body.
    
- **Cache-Control Header:** Tells intermediaries (like CDNs) and browsers how long they can safely cache a response without asking the server again (`Cache-Control: max-age=3600`).
