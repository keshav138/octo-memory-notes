Think of pagination as how you flip through a massive book. If the book has 10,000 pages, you need a strategy to find specific entries without getting lost or exhausted.

There are two primary ways to do this in a database: **Offset** (the traditional way) and **Cursor-based** (the "Indexed Seek" way).

---

## 1. Offset Pagination (The "Count and Skip" Method)

This is the most common method. You tell the database: "Skip the first 100 rows and give me the next 10."

- **The SQL:** `SELECT * FROM products ORDER BY id LIMIT 10 OFFSET 100;`
    
- **How it works:** The database starts at the very first row and counts 100 items. Once it reaches 101, it starts collecting data.
    

### The Problems:

1. **Performance:** As your "skip" number gets larger (e.g., `OFFSET 1,000,000`), the database still has to read all those million rows just to throw them away. This is slow and heavy on memory.
    
2. **Inaccuracy (The Drift):** If an item is deleted or added while a user is paginating, they might see the same item twice or skip an item entirely because the "positions" shifted.
    

---

## 2. Cursor-Based Pagination (The "Bookmark" Method)

Instead of telling the database how many rows to skip, you give it a "bookmark" from the last item you saw. This is where **Indexed Seek** comes in.

- **The SQL:** `SELECT * FROM products WHERE id > 500 ORDER BY id LIMIT 10;`
    
- **How it works:** You don't say "Skip 500." You say "Find the row where ID is 500, and give me the 10 rows immediately after it."
    

### Why it's better:

1. **Performance (Indexed Seek):** If your `id` column is indexed, the database doesn't "count" anything. It uses a **B-Tree search** (Indexed Seek) to jump directly to the exact spot in the data. It's just as fast for page 1,000,000 as it is for page 1.
    
2. **Consistency:** Even if rows are added or deleted "behind" you, your cursor (the ID) stays locked to a specific point in the data set. You won't see duplicates.
    

---

## 3. Comparison: Offset vs. Indexed Seek (Cursor)

|**Feature**|**Offset Pagination**|**Cursor / Indexed Seek**|
|---|---|---|
|**Speed**|Gets slower as you go deeper.|Stays consistently fast.|
|**Real-time Data**|Prone to duplicates/skipping.|Highly stable and accurate.|
|**Implementation**|Very easy (standard SQL).|Slightly harder (requires unique, sortable columns).|
|**Best For**|Small datasets, "Page 1, 2, 3" UI.|Infinite scrolls (Social Media feeds), massive APIs.|

---

## Summary of "Indexed Seek"

The term **Indexed Seek** simply refers to the database's ability to use an index to find a specific value instantly.

In **Offset**, the database does an **Index Scan** (reads everything until it hits the offset). In **Cursor**, the database does an **Index Seek** (jumps straight to the cursor value).

> **Pro-Tip:** If you are building a feed like Instagram or Twitter where users scroll endlessly, always use **Cursor-based** pagination. If you use Offset, your server will start sweating the moment a user scrolls to "Page 500."
