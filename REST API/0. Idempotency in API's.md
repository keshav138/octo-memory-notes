In the world of APIs, **idempotency** is a fancy way of saying: "No matter how many times you perform the same action, the end result on the server remains the same."

It’s a safety net for distributed systems. If a network hiccup happens and you aren't sure if your request went through, an idempotent method allows you to safely retry without accidentally doubling a charge or creating duplicate data.

---

## The Breakdown: Why the Difference?

To understand why some methods are idempotent and others aren't, we look at the intended side effects on the server's state.

### 1. PUT (Idempotent)

The `PUT` method is defined as a **replacement** operation. You are sending a full representation of a resource to a specific URL (e.g., `/users/123`).

- **First call:** Updates the user with ID 123 to "Alex."
    
- **Subsequent calls:** Updates the user with ID 123 to "Alex" again.
    
- **Result:** The state of the resource remains exactly the same after the first call. It doesn't matter if you hit the button 1 time or 100 times; Alex is still Alex.
    

### 2. DELETE (Idempotent)

This one often trips people up because the _response code_ might change, but the **server state** does not.

- **First call:** Deletes the resource. The resource is now gone.
    
- **Subsequent calls:** Tries to delete the resource. It's already gone.
    
- **Result:** The resource remains deleted. Whether you get a `204 No Content` the first time and a `404 Not Found` the second time, the permanent effect on the database is identical: the record is absent.
    

### 3. POST (NOT Idempotent)

`POST` is generally used for **creation** or processing where the server decides the URI of the new resource.

- **First call:** Creates "User A."
    
- **Second call:** Creates another "User A" (usually with a different unique ID).
    
- **Result:** Every time you send the request, you add a new entry to the database. If you retry a `POST` request because of a timeout, you risk creating duplicate orders, accounts, or comments.

## Comparison Table

|**Method**|**Idempotent?**|**Why?**|
|---|---|---|
|**GET**|Yes|It is "safe"; it only retrieves data and changes nothing.|
|**PUT**|**Yes**|It overwrites a specific target; repeating it changes nothing further.|
|**DELETE**|**Yes**|Once a resource is gone, it stays gone.|
|**POST**|**No**|It usually appends data; repeating it creates duplicates.|
|**PATCH**|No*|Usually not, as it can involve operations like "add 5 to the balance."|

> **A Note on PATCH:** While `PATCH` can be written to be idempotent (e.g., "Set age to 30"), it is technically classified as non-idempotent because it can be used for incremental updates that change the result every time they run.
